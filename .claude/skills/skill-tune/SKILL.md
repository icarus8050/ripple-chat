# skill-tune

스킬 라이브러리와 규칙을 지속 관리하기 위한 **메타 스킬**. 반복 이슈를 실행 가능한 규칙/체크리스트로 환원한다.

## 절차

1. **입력 수집** (이미 대화 맥락에 명확하면 생략)
   - 어떤 이슈/패턴인가? — 한두 문장.
   - 이번이 몇 번째 반복인가? **1회성이면 반영 금지** — 종료.
   - 어느 영역에 속하는가? (Reactive / WebSocket / 주석 / 테스트 / 패키지 / 빌드 / 워크플로우)

2. **환원 방식 결정**
   - **기존 스킬 보강**: 해당 스킬 체크리스트에 한 줄 추가. 기존 항목과 통합 가능하면 통합.
   - **CLAUDE.md 규칙 추가/수정**: 해당 섹션에 한 줄 추가. 금지/허용을 명확히.
   - **새 스킬 생성**: 패턴이 두 번 이상 반복됐고 기존 스킬 체크리스트로 흡수하기 어려울 때만. 단일 책임 유지.
   - **스킬 폐기**: 스택/패턴 변경으로 무의미해진 스킬은 삭제 (남기면 잘못된 컨텍스트 주입).

3. **제안 → 확인 → 적용**
   - 변경 diff를 먼저 제시한다. 사용자 승인 전에는 파일을 수정하지 않는다.
   - 여러 파일 변경이 동시에 필요하면 나눠서 제시.

4. **커밋 분리**
   - 별도 커밋. 메시지 prefix:
     - 스킬 파일만 변경: `chore(skills): ...`
     - CLAUDE.md 규칙 변경: `docs: ...`
   - 스킬과 규칙을 **같은 커밋에 섞지 않는다**.

## 규칙

- **1회성 이슈로 반영하지 않는다** — 과적합 방지.
- 체크리스트는 **짧게**. 스킬 한 개 프롬프트가 지나치게 길어지면 (> 약 60줄) 분리 혹은 재작성 고려.
- 스킬 이름은 동사/명령형 kebab-case (`decision-log`, `reactive-review`, `self-review`).
- 이 스킬(`skill-tune`) 자체를 고칠 때도 같은 절차 — diff 제시 → 승인 → 커밋 분리.
- 스킬 변경 결과는 다음 루프의 `/self-review` 가 실제로 잡아내는지 한 번 검증한 뒤 종료.
